# 时间精确度配置文档

## 概述

SCIM Go 服务支持可配置的时间精确度功能，允许管理员根据业务需求调整系统中时间字段的精确度级别。该配置影响所有涉及时间处理的模块，包括数据存储、API 接口响应和日志记录。

## 配置项说明

### 配置位置

时间精确度配置位于 `config.yaml` 文件的 `time_precision` 部分：

```yaml
# 时间精确度配置
time_precision:
  # 时间精确度级别
  level: "second"
  # 自定义时间格式（可选）
  format: ""
```

### 配置项详解

#### 1. level - 时间精确度级别

**配置名称**: `time_precision.level`  
**环境变量**: `SCIM_TIME_PRECISION_LEVEL`  
**数据类型**: 字符串  
**默认值**: `second`

**可选值**:

| 级别值 | 说明 | 格式示例 |
|--------|------|----------|
| `second` | 秒级精确度（默认） | `2026-03-05 14:30:00` |
| `minute` | 分钟级精确度 | `2026-03-05 14:30` |
| `millisecond` | 毫秒级精确度 | `2026-03-05 14:30:00.123` |
| `microsecond` | 微秒级精确度 | `2026-03-05 14:30:00.123456` |
| `nanosecond` | 纳秒级精确度 | `2026-03-05 14:30:00.123456789` |

#### 2. format - 自定义时间格式

**配置名称**: `time_precision.format`  
**环境变量**: `SCIM_TIME_PRECISION_FORMAT`  
**数据类型**: 字符串  
**默认值**: `""`（空，使用级别对应的默认格式）

**说明**: 
- 当设置自定义格式时，将优先使用自定义格式
- 格式使用 Go 语言的 time 包格式（如：`2006-01-02 15:04:05`）
- 建议仅在需要特殊格式时使用，通常使用 `level` 配置即可

## 配置生效范围

### 1. API 接口响应

时间精确度配置会影响以下 API 响应字段的时间格式：

- `meta.created` - 资源创建时间
- `meta.lastModified` - 资源最后修改时间

**示例**:

秒级精确度响应：
```json
{
  "meta": {
    "created": "2026-03-05T14:30:00+08:00",
    "lastModified": "2026-03-05T14:30:00+08:00"
  }
}
```

毫秒级精确度响应：
```json
{
  "meta": {
    "created": "2026-03-05T14:30:00.123+08:00",
    "lastModified": "2026-03-05T14:30:00.456+08:00"
  }
}
```

### 2. 数据存储

数据库中的时间字段存储格式不受此配置影响，始终使用数据库原生时间类型存储完整时间戳。该配置仅影响时间的外部表示格式。

### 3. 日志记录

日志中的时间戳格式会根据配置的时间精确度级别自动调整：

- 秒级：`2026-03-05 14:30:00`
- 毫秒级：`2026-03-05 14:30:00.123`

## 配置方法

### 方法一：配置文件（推荐）

编辑 `config.yaml` 文件：

```yaml
time_precision:
  level: "millisecond"
```

### 方法二：环境变量

```bash
export SCIM_TIME_PRECISION_LEVEL=millisecond
```

### 方法三：命令行参数

```bash
go run main.go -time-precision-level=millisecond
```

**配置优先级**: 命令行参数 > 环境变量 > 配置文件 > 默认值

## 配置变更与历史数据

### 向后兼容性

- 配置变更后，系统能够正确解析历史数据中的时间格式
- 支持解析多种时间格式（ISO 8601 标准格式）
- 新旧数据的时间格式一致性由系统自动处理

### 动态调整

- 修改配置后需要重启服务才能生效
- 建议在维护窗口期间进行配置变更
- 变更后新创建的数据将使用新的时间格式

## 最佳实践

### 1. 选择合适的精确度级别

- **秒级（默认）**: 适用于大多数业务场景，满足 SCIM 2.0 标准要求
- **毫秒级**: 适用于需要更高时间精度的场景，如审计日志
- **微秒/纳秒级**: 仅在特殊需求下使用，可能增加数据传输量

### 2. 生产环境建议

- 在测试环境充分验证后再应用到生产环境
- 记录配置变更历史，便于问题追溯
- 监控 API 响应时间，评估高精确度对性能的影响

### 3. 客户端兼容性

- 确保客户端能够处理不同精确度的时间格式
- 建议客户端使用标准 ISO 8601 时间解析库
- 避免客户端对时间格式进行硬编码解析

## 故障排除

### 常见问题

#### Q1: 配置未生效

**可能原因**:
- 配置文件格式错误
- 配置项名称拼写错误
- 未重启服务

**解决方法**:
1. 检查 YAML 格式是否正确
2. 确认配置项名称为 `time_precision.level`
3. 重启服务使配置生效

#### Q2: 历史数据时间格式不一致

**说明**: 这是正常现象，系统会自动处理不同格式的时间解析

**解决方法**: 无需处理，系统支持向后兼容的时间解析

#### Q3: 客户端解析时间失败

**可能原因**: 客户端未正确处理 ISO 8601 时间格式

**解决方法**:
1. 更新客户端使用标准时间解析库
2. 确保客户端支持带时区的时间格式
3. 考虑在 API 网关层统一时间格式
